# Governance 🦀

## How Decisions Get Made

Pincer is a young project. We believe governance should evolve with the community, not be imposed prematurely. What follows is honest about where we are today and transparent about where we're heading.

---

## Current State: Benevolent Dictator

Pincer is currently maintained and led by its original creator. This is normal for early open-source projects, and we're not pretending otherwise.

**What this means in practice:**

- The maintainer has final say on merges, releases, and roadmap direction
- All decisions are made in public (GitHub Issues, Discussions, Discord) — nothing happens in private channels
- The maintainer can be wrong, and community feedback changes decisions regularly
- The maintainer's time is finite, and community help is essential, not decorative

**What this does NOT mean:**

- That other opinions don't matter (they do, and they shape the project daily)
- That the maintainer can't be challenged (please do — respectfully)
- That this structure is permanent (it will evolve as the community grows)

## Decision Categories

### Tier 1: Just Do It

No approval needed. Open a PR.

- Bug fixes
- Typo corrections
- Documentation improvements
- Test coverage additions
- Dependency updates (non-breaking)
- New skills that follow the skill guide

### Tier 2: PR Review

Standard PR review by a maintainer.

- New features within existing architecture
- New bundled skills with external dependencies
- Refactoring that doesn't change behavior
- CI/CD changes
- Dashboard improvements

### Tier 3: Discussion First

Open a GitHub Discussion before writing code.

- New channel integrations
- Changes to the security model
- Breaking API changes
- Changes to the skill manifest format
- New dependencies that affect the install size
- Architecture changes that affect multiple modules

### Tier 4: Community Input

Major decisions that affect the direction of the project.

- Roadmap priorities
- Licensing changes
- Branding changes
- Whether to accept corporate sponsorship
- Adding or removing maintainers
- Major version releases

These decisions are discussed openly in GitHub Discussions with a minimum 7-day comment period before action.

## Path to Maintainership

As the community grows, we'll add maintainers. Here's how that will work:

### What We Look For

We don't have a checklist. We look for people who consistently demonstrate:

- **Good judgment** — they review PRs thoughtfully, not just for correctness but for fit
- **Reliability** — they show up, follow through, and communicate proactively
- **Kindness** — they make the community better just by being in it
- **Technical skill** — they understand the codebase well enough to spot problems and suggest improvements
- **Security mindset** — they think about threat models, not just functionality

### What Maintainers Do

- Review and merge PRs
- Triage issues
- Help contributors get unstuck
- Participate in architecture discussions
- Release new versions
- Handle security reports
- Moderate community spaces

### How It Happens

1. The project lead identifies candidates based on sustained, high-quality contributions
2. A private conversation to gauge interest and availability
3. Public announcement in Discord and GitHub Discussions
4. Gradual increase in responsibilities (start with specific areas, expand over time)

There's no formal application process because titles should follow work, not the other way around. If you're doing the work of a maintainer, the title will follow.

## Financial Transparency

If and when Pincer receives sponsorship, donations, or any form of funding:

- All income and expenses will be publicly documented
- Funds will be used for: infrastructure costs, security audits, bounties, and contributor stipends
- The project lead will publish quarterly financial summaries
- Major spending decisions (>$500) will be discussed with maintainers before commitment

Currently, Pincer has no funding and no financial obligations. The maintainer donates their time.

---

## Evolving This Document

This governance document will change as the community grows. Changes follow Tier 4 (Community Input) process: proposed in GitHub Discussions, 7-day comment period, maintainer decision incorporating community feedback.

The goal is always the same: make decisions transparently, include the people affected, and keep the process as lightweight as the community allows.
